## 核心知识点

1. 架构设计先回答瓶颈在哪里，再谈技术名词。
2. 容量评估、压测、降级预案要同时存在。
3. 演进方案需要兼顾收益、复杂度和交付节奏。

## 完整案例 Walkthrough（L2/L3 深挖）

### 场景输入

- 业务从日均 100 万请求增长到 1000 万，请求峰值明显上升。

### 线上现象

- 核心链路在活动期间出现扩容滞后和局部热点。

### 证据采集

- 分析流量模型、容量基线、热点分布、节点利用率和降级触发记录。

### 定位分析

- 定位为容量估算偏保守 + 热点隔离不足 + 扩容策略滞后。

### 修复动作

- 重建容量模型，拆分热点链路，提前扩容并完善自动降级策略。

### 回归验证

- 通过压测和演练验证峰值吞吐、可用性和恢复时长指标。

### 实战排障清单

- 架构改造先明确业务目标和约束。
- 每个方案要给出回滚路径。
- 容量评估必须与压测和监控联动。


## 错答示例 -> 修正答法 -> 打分差异（章级题解）

### 练习题目（围绕本章：团队工程规范落地）

- 请用 90 秒说明“定义 -> 原理 -> 场景 -> 风险 -> 验证”完整答题链路。
- 请补充至少 1 个线上或项目中的落地例子，并说明为什么这样做。

### 常见错答示例（低分版）

- 只说概念，不说机制：例如只背定义，无法解释底层流程。
- 只说优点，不说边界：没有说明适用条件与失败场景。
- 没有指标验证：讲完方案后不给量化结果或回归口径。

### 修正答法（高分版）

1. 先给结论：一句话说清本章知识点解决什么问题。
2. 再讲原理：用 2~3 个关键机制串起完整流程。
3. 再落场景：给出一个可复现的业务场景和方案选择理由。
4. 再说风险：列出至少 2 个常见坑和对应防护动作。
5. 最后验证：给出可观测指标（如延迟、错误率、吞吐、资源占用）与目标阈值。

### 打分差异示例（同题对比）

| 评分维度 | 错答（低分） | 修正（高分） | 提升点 |
|---|---|---|---|
| 概念准确 | 只背术语 | 术语 + 边界条件 | 避免概念混淆 |
| 原理完整 | 断点式描述 | 链路化描述 | 解释能力更强 |
| 场景匹配 | 空泛举例 | 贴近业务约束 | 方案更可信 |
| 风险意识 | 不提失败 | 提供兜底与回滚 | 工程可落地 |
| 验证闭环 | 无量化指标 | 指标 + 阈值 + 回归 | 可复盘可验收 |

## Java 示例代码（含注释，可直接运行）



**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

**预期输出（示例）：**
```text
nodes=6
```

```java
public class Main {
    public static void main(String[] args) {
        long peakQps = 5000;
        int singleNodeQps = 1200;
        double redundancy = 1.3;
        // 容量估算 = 峰值QPS / 单机能力 * 冗余系数
        int nodes = (int) Math.ceil((peakQps / (double) singleNodeQps) * redundancy);
        System.out.println("nodes=" + nodes);
    }
}
```
