# L3-03 高可用与稳定性工程

## 这是什么

高可用不是“服务不挂”，而是“挂了也有秩序地退化并快速恢复”。

核心内容：
- 多副本和容灾
- 故障隔离与降级
- 应急机制和演练

## 稳定性体系图

```mermaid
flowchart LR
    A[预防: 容量+压测] --> B[保护: 限流/熔断/隔离]
    B --> C[恢复: 回滚/切流/降级]
    C --> D[复盘: 根因分析与治理]
```

## 核心实践

### 1) 故障隔离

- 线程池隔离、服务隔离、机房隔离。
- 避免单点故障向全链路扩散。

### 2) 降级策略

- 优先保核心交易路径。
- 非核心功能可降级为兜底数据或稍后重试。

### 3) 演练机制

- 定期故障演练（网络抖动、依赖超时、节点宕机）。
- 演练要有回放数据和恢复指标。

## 高频面试题

### Q1：系统可用性怎么提升？

答题骨架：
1. 先识别单点（服务、数据库、缓存）。
2. 通过冗余和故障转移提升可用。
3. 通过限流熔断控制故障半径。
4. 通过监控告警和演练缩短恢复时间。

### Q2：线上事故复盘应包含什么？

答题骨架：
1. 事件时间线。
2. 根因分析。
3. 影响范围和损失评估。
4. 永久治理项和负责人截止日期。

## 完整案例 Walkthrough（L2/L3 深挖）

### 场景输入

- 业务从日均 100 万请求增长到 1000 万，请求峰值明显上升。

### 线上现象

- 核心链路在活动期间出现扩容滞后和局部热点。

### 证据采集

- 分析流量模型、容量基线、热点分布、节点利用率和降级触发记录。

### 定位分析

- 定位为容量估算偏保守 + 热点隔离不足 + 扩容策略滞后。

### 修复动作

- 重建容量模型，拆分热点链路，提前扩容并完善自动降级策略。

### 回归验证

- 通过压测和演练验证峰值吞吐、可用性和恢复时长指标。

### 实战排障清单

- 架构改造先明确业务目标和约束。
- 每个方案要给出回滚路径。
- 容量评估必须与压测和监控联动。

## Java 示例代码（含注释，可直接运行）



**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

**预期输出（示例）：**
```text
nodes=6
```

```java
public class Main {
    public static void main(String[] args) {
        long peakQps = 5000;
        int singleNodeQps = 1200;
        double redundancy = 1.3;
        // 容量估算 = 峰值QPS / 单机能力 * 冗余系数
        int nodes = (int) Math.ceil((peakQps / (double) singleNodeQps) * redundancy);
        System.out.println("nodes=" + nodes);
    }
}
```
