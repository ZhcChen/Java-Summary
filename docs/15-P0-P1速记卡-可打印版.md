# P0 / P1 速记卡（可打印版）

> 用法：每次抽 5 张卡，按“30 秒结论 + 60 秒展开 + 30 秒场景”作答。

## P0 必问题卡

### [L1-P0-01] Java 为什么只有值传递？
- 结论：参数传递本质都是值拷贝，引用类型拷贝的是地址值。
- 展开：可修改对象状态；不能改变外部引用指向。
- 场景：解释方法参数修改对象与重新赋值的差异。

### [L1-P0-02] `HashMap` 为什么线程不安全？
- 结论：并发 `put/resize` 存在竞态，可能数据覆盖或状态不一致。
- 展开：缺少同步控制；扩容迁移期间问题更明显。
- 场景：并发场景替换为 `ConcurrentHashMap`。

### [L1-P0-03] `synchronized` 和 `volatile` 区别？
- 结论：`synchronized` 提供互斥+可见性；`volatile` 只保证可见性和有序性。
- 展开：`volatile` 不能保证复合操作原子性（如 `i++`）。
- 场景：状态标记用 `volatile`，临界区保护用 `synchronized`。

### [L1-P0-04] JVM 内存区域有哪些？
- 结论：堆、栈、方法区（元空间）、程序计数器、本地方法栈。
- 展开：对象主要在堆，栈存方法帧和局部变量。
- 场景：区分堆 OOM 与栈溢出定位方向。

### [L1-P0-05] MySQL 索引为什么能加速查询？
- 结论：索引通过有序结构减少扫描行数。
- 展开：B+Tree 支持高效范围查询和磁盘页局部性。
- 场景：用 `EXPLAIN` 验证是否命中索引。

### [L1-P0-06] Redis 为什么快？
- 结论：内存访问 + 高效数据结构 + 单线程事件驱动（经典模型）。
- 展开：网络 IO 模型和避免锁竞争降低开销。
- 场景：热点数据缓存，降低数据库压力。

### [L2-P0-01] 线程池参数怎么配？
- 结论：先按任务类型给初值，再用压测和指标迭代。
- 展开：关注核心线程、最大线程、队列容量、拒绝策略。
- 场景：CPU 密集与 IO 密集线程数策略不同。

### [L2-P0-02] Full GC 频繁怎么排查？
- 结论：先采集证据（GC 日志/堆快照/线程栈）再定位根因。
- 展开：分清分配速率问题、晋升问题、泄漏问题。
- 场景：给出短期止血（降流量）+长期治理（对象模型优化）。

### [L2-P0-03] 如何保证消息不丢失？
- 结论：生产端确认 + Broker 持久化 + 消费端确认重试。
- 展开：三段任何一段缺失都可能丢消息。
- 场景：关键业务增加死信队列和补偿任务。

### [L2-P0-04] 缓存和数据库一致性怎么保证？
- 结论：常见策略是先更新 DB，再删除缓存。
- 展开：必要时配合延迟双删或订阅 binlog。
- 场景：读多写少场景采用最终一致方案。

### [L3-P0-01] 如何设计秒杀系统？
- 结论：流量前置削峰、库存原子扣减、下单异步化、失败补偿。
- 展开：核心是防超卖、防击穿、防雪崩。
- 场景：限流+令牌+MQ 下单链路。

### [L3-P0-02] 分布式事务方案怎么选？
- 结论：先定义一致性等级，再选 2PC/TCC/SAGA/本地消息表。
- 展开：一致性越强成本越高，侵入性越大。
- 场景：资金类偏 TCC，长流程偏 SAGA。

## P1 高频进阶卡

### [L1-P1-01] `ArrayList` vs `LinkedList` 怎么选？
- 结论：默认优先 `ArrayList`，少数频繁中间插入再考虑 `LinkedList`。
- 展开：缓存局部性常让 `ArrayList` 更快。
- 场景：先看读写比例再选。

### [L1-P1-02] 为什么金额用 `BigDecimal`？
- 结论：避免二进制浮点误差，保证十进制精确计算。
- 展开：构造优先字符串，除法指定舍入模式。
- 场景：订单金额、折扣计算、对账。

### [L2-P1-01] 为什么不建议直接用 `Executors`？
- 结论：默认工厂可能产生无界队列/无限线程风险。
- 展开：生产环境要显式定义队列上限和拒绝策略。
- 场景：高峰流量下避免 OOM 和雪崩。

### [L2-P1-02] 索引失效常见原因有哪些？
- 结论：函数操作列、隐式类型转换、最左前缀不满足等。
- 展开：失效本质是优化器无法高效利用索引。
- 场景：慢 SQL 排查先查执行计划。

### [L2-P1-03] 缓存穿透/击穿/雪崩怎么区分？
- 结论：穿透=不存在数据，击穿=热点 key 过期，雪崩=大量 key 同时失效。
- 展开：分别对应布隆过滤、互斥重建、随机过期+限流。
- 场景：大促和热点活动前必做治理。

### [L2-P1-04] 如何做限流熔断降级联动？
- 结论：入口限流、依赖熔断、结果降级三层联动。
- 展开：目标是控制故障半径和恢复时间。
- 场景：下游超时时快速失败并返回兜底。

### [L2-P1-05] Trace/Logs/Metrics 怎么协同？
- 结论：统一 traceId，先看指标再追踪链路最后查日志。
- 展开：三者分别回答“是否异常、在哪异常、为何异常”。
- 场景：线上慢请求定位。

### [L3-P1-01] 容量评估怎么做？
- 结论：峰值流量模型 + 单机压测能力 + 冗余系数。
- 展开：计算只是起点，必须结合压测校验。
- 场景：节日前扩容评估。

### [L3-P1-02] CAP 在项目里怎么落地？
- 结论：分区容错前提下，一致性与可用性按业务选择。
- 展开：交易核心偏一致，查询系统偏可用。
- 场景：订单状态最终一致性设计。

### [L3-P1-03] 本地消息表适合什么场景？
- 结论：适合最终一致、可容忍短时延迟的业务流程。
- 展开：事务内落库 + 异步投递 + 补偿重试。
- 场景：订单创建后异步通知。

### [L3-P1-04] 如何做故障复盘闭环？
- 结论：时间线、根因、影响、治理项、负责人、截止日期缺一不可。
- 展开：复盘重点是防复发，不是“找人背锅”。
- 场景：线上事故后 48 小时内出复盘报告。

### [L3-P1-05] 架构决策如何表达更有说服力？
- 结论：按“背景-约束-方案对比-取舍-结果”表达。
- 展开：用数据支撑取舍，不堆技术名词。
- 场景：系统设计面和晋升答辩。
