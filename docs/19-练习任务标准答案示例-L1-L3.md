# 练习任务标准答案示例（L1/L2/L3）

> 目的：把“会看懂”升级为“会输出”，给每个层级提供可复用的答题模板与评分标准。

## 使用方式

1. 先完成对应章节里的“练习任务”，不要先看答案。
2. 用本页“标准答案结构”做自评，缺哪块补哪块。
3. 最后用“评分标准”打分，低于 80 分需要重做。

## 通用答题结构（建议 5 段）

1. **问题定义**：这道题在解决什么业务问题。
2. **核心原理**：关键机制、数据结构或流程。
3. **方案选择**：为什么选 A，不选 B（权衡点）。
4. **风险与兜底**：可能踩坑、监控指标、回滚方案。
5. **落地结果**：如何验证效果（指标前后对比）。

## 通用评分标准（100 分）

| 维度 | 分值 | 合格线 | 说明 |
|---|---:|---:|---|
| 概念准确 | 25 | 20 | 术语与定义是否正确 |
| 原理完整 | 25 | 18 | 是否讲清关键流程与边界 |
| 场景匹配 | 20 | 15 | 方案是否贴合业务约束 |
| 风险意识 | 15 | 10 | 是否考虑失败与补偿 |
| 表达结构 | 15 | 10 | 是否按结构清晰输出 |

## L1 标准答案示例

### 示例 1：`HashMap` 与 `ConcurrentHashMap` 如何选择？

**标准答案示例：**

- 定义：`HashMap` 线程不安全，`ConcurrentHashMap` 在并发场景下提供线程安全读写。
- 原理：JDK8 的 `ConcurrentHashMap` 采用 `CAS + synchronized`，通过桶级别同步降低锁竞争。
- 选择：单线程或只读场景用 `HashMap`；多线程共享写入必须用 `ConcurrentHashMap`。
- 风险：在高并发下仍可能出现热点桶竞争，需要关注冲突率与扩容成本。
- 验证：压测对比吞吐、P99 与 CPU，确认并发安全与性能是否满足目标。

### 示例 2：`BigDecimal` 为什么必须用字符串构造？

**标准答案示例：**

- 定义：`double` 是二进制浮点，表示十进制金额会有精度误差。
- 原理：`new BigDecimal("0.1")` 可精确表达十进制；`new BigDecimal(0.1)` 会把二进制误差带进来。
- 选择：金额、税率、折扣等金融值统一 `BigDecimal`，并显式指定舍入规则。
- 风险：未指定舍入模式时除法可能抛异常；比较时应使用 `compareTo` 而不是 `equals`。
- 验证：用 0.1 + 0.2、除法场景编写单元测试校验结果。

## L2 标准答案示例

### 示例 1：线程池任务堆积如何定位与调优？

**标准答案示例：**

- 定义：请求量增长导致任务提交速度高于消费速度，队列持续增长。
- 原理：线程池吞吐受 `corePoolSize/maxPoolSize/queueCapacity/rejectPolicy` 共同影响。
- 选择：优先按压测数据调整核心线程与队列；对慢接口做线程池隔离。
- 风险：盲目增大队列会拉高延迟；盲目增线程会导致上下文切换与 CPU 抖动。
- 验证：观察 `activeCount/queueSize/rejectCount` 与 P99 是否回落到阈值内。

### 示例 2：线上 OOM 的标准定位流程是什么？

**标准答案示例：**

- 定义：JVM 无法再分配内存，常见为堆、元空间、直接内存耗尽。
- 原理：不同区域 OOM 对应不同根因，如对象泄漏、类动态生成过多、`ByteBuffer` 未释放。
- 选择：先定位区域，再结合 dump、GC 日志、对象引用链定位根因。
- 风险：只调大堆不查根因会复发；全量 dump 可能加重线上压力。
- 验证：修复后压测与灰度，确认 Full GC 频次、老年代占用、错误率恢复。

### 示例 3：缓存一致性如何落地？

**标准答案示例：**

- 定义：数据库与缓存在更新时可能出现短暂不一致。
- 原理：常用“先更新 DB，再删缓存”的 Cache Aside；配合重试与延迟双删降低脏读窗口。
- 选择：强一致要求极高时可引入消息驱动的异步补偿。
- 风险：并发写、网络抖动、消息丢失会导致不一致扩大。
- 验证：构造并发更新场景，统计脏读率并验证补偿命中率。

## L3 标准答案示例

### 示例 1：容量估算如何从业务指标推导机器数？

**标准答案示例：**

- 定义：根据目标 QPS、平均耗时、冗余系数估算资源规模。
- 原理：并发数约等于 `QPS x RT`，再按单机吞吐与安全冗余折算实例数。
- 选择：按峰值流量而非平均流量规划，并预留 20%~30% 弹性。
- 风险：只看 CPU 不看下游容量会出现“本服务健康、全链路超时”。
- 验证：压测峰值 + 故障演练，确认 P99、错误率、降级触发率。

### 示例 2：TCC 与 SAGA 怎么选？

**标准答案示例：**

- 定义：两者都用于跨服务最终一致，但代价与适用场景不同。
- 原理：TCC 显式预留资源、强业务侵入；SAGA 通过前向事务 + 补偿事务完成一致。
- 选择：资金冻结类强约束优先 TCC；长流程、低耦合业务优先 SAGA。
- 风险：补偿失败会导致长尾脏数据，需要幂等、重试、人工兜底。
- 验证：用“成功/超时/部分失败”三类链路进行演练并记录补偿成功率。

## Java 评分器示例（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
import java.util.LinkedHashMap;
import java.util.Map;

public class Main {
    public static void main(String[] args) {
        // 每个维度按 0~满分打分
        Map<String, Integer> scores = new LinkedHashMap<>();
        scores.put("概念准确", 22);
        scores.put("原理完整", 20);
        scores.put("场景匹配", 17);
        scores.put("风险意识", 12);
        scores.put("表达结构", 13);

        int total = 0;
        for (int score : scores.values()) {
            total += score;
        }

        System.out.println("总分: " + total);
        System.out.println(total >= 80 ? "结论: 达到合格线，可进入下一题。" : "结论: 低于合格线，需要重做并补充薄弱项。");
    }
}
```

**预期输出示例：**

```text
总分: 84
结论: 达到合格线，可进入下一题。
```
