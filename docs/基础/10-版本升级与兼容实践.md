# 10 版本升级与兼容实践

## 本章目标

- 掌握 Java 版本升级的最小可行流程。
- 能识别升级中的兼容风险与验证重点。
- 形成“可灰度、可回滚”的升级策略。

## 前置知识

- 了解项目构建、测试、发布的基本流程。
- 对 Java 版本特性有基本认知。

## 核心概念

1. 升级不是一次性动作，而是分阶段工程治理。
2. 兼容风险包括编译、依赖、运行时行为和性能。
3. 回归测试与灰度验证决定升级成败。

## 原理展开

- 统一本地、CI、生产的 JDK 版本是前提。
- 依赖库升级要先做兼容矩阵，再做代码改造。
- 性能验证必须比较升级前后指标，不只看功能通过。

## 典型场景

- 旧服务从 Java 8 升级到 17/21/25。
- 核心依赖（Spring、驱动、监控 SDK）兼容性升级。
- 容器镜像 JDK 更换带来的行为变化验证。

## 常见误区

1. 只改构建版本，不做运行时验证。
2. 功能测试通过就直接上线，不看性能指标。
3. 缺少回滚方案，升级失败时恢复缓慢。

## Java 示例代码（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
import java.util.LinkedHashMap;
import java.util.Map;

public class Main {
    public static void main(String[] args) {
        // 有序清单便于发布评审逐项确认
        Map<String, Boolean> checklist = new LinkedHashMap<>();
        checklist.put("统一JDK版本", true);
        checklist.put("依赖兼容性检查", true);
        checklist.put("回归测试通过", false);
        checklist.put("压测指标达标", false);

        long passed = checklist.values().stream().filter(Boolean::booleanValue).count();
        System.out.println("passed=" + passed + "/" + checklist.size());

        checklist.forEach((item, ok) ->
                System.out.println(item + " -> " + (ok ? "OK" : "TODO"))
        );

        boolean ready = passed == checklist.size();
        System.out.println("canRelease=" + ready);
    }
}
```

**预期输出示例：**

```text
passed=2/4
统一JDK版本 -> OK
依赖兼容性检查 -> OK
回归测试通过 -> TODO
压测指标达标 -> TODO
canRelease=false
```

## 练习与自测

1. 写一份“8 -> 21/25 升级检查清单”。
2. 设计灰度验证指标（延迟、错误率、CPU、内存）。
3. 自测：能否说清“升级失败时的回滚动作顺序”。

## 本章小结

- 版本升级是系统工程，关键在流程和验证。
- 没有验证数据的升级结论，不可靠。
