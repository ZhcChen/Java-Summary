# 10 版本升级与兼容实践

## 本章目标

- 从工程角度理解 Java 升级的主要风险。
- 掌握最小可行的升级流程。
- 建立“可回滚、可验证”的升级策略。

## 一、升级中最常见的问题

1. JDK 版本不一致（本地、CI、生产不同）。
2. 依赖库与新 JDK 不兼容。
3. 升级后功能正常但性能回退。

## 二、升级流程（建议）

1. 统一基线：先统一构建 JDK 与运行 JDK。
2. 依赖盘点：确认框架和中间件支持目标版本。
3. 编译验证：修复语法与 API 变更。
4. 回归测试：核心链路 + 异常链路。
5. 压测验证：延迟、吞吐、内存、GC 指标对比。
6. 灰度发布：小流量观察后再全量。

## 三、兼容策略

- 新特性采用“渐进启用”，避免一次性大改。
- 关键模块保留回滚方案。
- 重要配置变更做好版本化记录。

## 四、练习任务

1. 为一个现有项目写“Java 8 -> 17”升级检查清单。
2. 设计一份灰度验证表：指标、阈值、回滚条件。
3. 输出一份升级复盘模板（问题、原因、改进）。

## Java 示例代码（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
import java.util.LinkedHashMap;
import java.util.Map;

public class Main {
    public static void main(String[] args) {
        // 用有序 Map 模拟升级检查项与状态
        Map<String, Boolean> checklist = new LinkedHashMap<>();
        checklist.put("统一JDK版本", true);
        checklist.put("依赖兼容性检查", true);
        checklist.put("回归测试通过", false);
        checklist.put("压测指标达标", false);

        long passed = checklist.values().stream().filter(Boolean::booleanValue).count();
        System.out.println("passed=" + passed + "/" + checklist.size());

        checklist.forEach((item, ok) ->
                System.out.println(item + " -> " + (ok ? "OK" : "TODO"))
        );

        boolean ready = passed == checklist.size();
        System.out.println("canRelease=" + ready);
    }
}
```

**预期输出示例：**

```text
passed=2/4
统一JDK版本 -> OK
依赖兼容性检查 -> OK
回归测试通过 -> TODO
压测指标达标 -> TODO
canRelease=false
```
