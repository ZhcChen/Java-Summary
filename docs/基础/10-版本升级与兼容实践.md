# 10 版本升级与兼容实践

## 本章目标

- 建立可执行的 Java 版本升级流程。
- 能识别兼容性风险并设计验证方案。
- 形成“可灰度、可观测、可回滚”的升级方法。

## 前置知识

- 了解基础构建、测试、发布流程。
- 已阅读 Java 9~25 特性章节。

## 一、升级是工程治理，不是改一个版本号

版本升级通常包含三层：

1. **语言层**：语法和编译器行为变化。
2. **库层**：依赖（框架、驱动、SDK）兼容性。
3. **运行层**：JVM 参数、容器镜像、监控告警行为。

只改 `pom`/`gradle` 版本而不做运行验证，风险很高。

## 二、推荐升级流程（最小可行）

1. **盘点现状**：当前 JDK、依赖、构建链路。
2. **建立兼容矩阵**：列出核心依赖在目标 JDK 下的支持情况。
3. **双环境验证**：旧版本与新版本并行跑回归测试。
4. **灰度发布**：小流量观察关键指标。
5. **全量切换 + 回滚预案**：明确“触发回滚阈值”。

## 三、验证指标建议（至少覆盖）

- 功能：接口正确率、核心场景通过率。
- 性能：P95/P99 延迟、吞吐、CPU、内存。
- 稳定性：错误率、超时率、重启次数。
- JVM：GC 次数与停顿时长。

## Java 示例代码（升级检查清单模拟）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
import java.util.LinkedHashMap;
import java.util.Map;

public class Main {
    public static void main(String[] args) {
        // 读取当前运行时版本（兼容多版本写法）
        String javaVersion = System.getProperty("java.version");
        System.out.println("runtimeJavaVersion=" + javaVersion);

        // 升级检查清单：每一项都应该有负责人和验收标准
        Map<String, Boolean> checklist = new LinkedHashMap<>();
        checklist.put("构建环境JDK已统一", true);
        checklist.put("核心依赖兼容性已确认", true);
        checklist.put("回归测试通过", false);
        checklist.put("压测指标达标", false);
        checklist.put("灰度与回滚预案已演练", false);

        long passed = checklist.values().stream().filter(Boolean::booleanValue).count();
        long total = checklist.size();

        checklist.forEach((item, ok) ->
                System.out.println(item + " -> " + (ok ? "OK" : "TODO"))
        );

        boolean canRelease = passed == total;
        System.out.println("checkPassed=" + passed + "/" + total);
        System.out.println("canRelease=" + canRelease);

        if (!canRelease) {
            // 在真实流程中，这里应阻断发布而不是“带病上线”
            System.out.println("action=继续补齐验证项，禁止直接发布");
        }
    }
}
```

**预期输出示例：**

```text
runtimeJavaVersion=...
构建环境JDK已统一 -> OK
核心依赖兼容性已确认 -> OK
回归测试通过 -> TODO
压测指标达标 -> TODO
灰度与回滚预案已演练 -> TODO
checkPassed=2/5
canRelease=false
action=继续补齐验证项，禁止直接发布
```

## 常见误区与修正

1. **误区：功能回归过了就可以发**  
   修正：性能与稳定性指标同样必须验收。

2. **误区：升级失败再临时回滚**  
   修正：回滚方案应在升级前预演。

3. **误区：升级只由开发负责**  
   修正：需要研发、测试、运维协同。

## 面试高频问答

### Q1：8 升 17/21/25 的关键风险是什么？

- 一句话结论：依赖兼容、运行时行为变化、性能波动是三大核心风险。
- 关键点：
  1. 框架与驱动版本可能不兼容。
  2. JVM 默认参数行为可能变化。
  3. 需要压测对比而不是主观判断。

### Q2：如何设计升级回滚策略？

- 一句话结论：双版本可切换 + 指标阈值触发 + 自动化回退流程。
- 关键点：
  1. 预先定义回滚阈值。
  2. 保留旧镜像与旧配置。
  3. 回滚后立即复盘根因。

## 练习与自测

1. 为自己的项目写一份升级检查表（至少 10 条）。
2. 定义三类回滚触发阈值（错误率、延迟、资源）。
3. 用 5 分钟口述你会如何执行一次“先灰度后全量”的升级。

## 本章小结

- 升级成功依赖流程化验证，而不是经验判断。
- 没有验证数据支撑的升级结论，都不可靠。
