# 04 异常机制与调试思维

## 本章目标

- 理解 Java 异常体系与业务异常边界。
- 写出可追踪、可定位、可恢复的异常处理逻辑。
- 形成“先复现、再定位、再修复、再验证”的调试闭环。

## 前置知识

- 会写 `try-catch-finally`。
- 能阅读堆栈信息中的类名、方法名和行号。

## 一、异常不是“报错结束”，而是“错误语义传递”

### 1) 两类异常的职责

- **Checked Exception**：编译期强制处理，常见于 IO、数据库、网络等外部资源。
- **Runtime Exception**：运行期错误，常见于参数非法、状态非法、代码缺陷。

### 2) 业务代码中的边界原则

1. 在“业务边界层”把底层异常转换为业务异常。
2. 在“入口层”统一兜底，避免异常直接泄露给调用方。
3. 异常信息应包含可定位字段（请求 ID、业务主键、错误码）。

## 二、调试四步法（建议固定流程）

1. **先复现**：确认触发条件，避免“猜测式修复”。
2. **看现场**：日志、堆栈、输入参数、最近变更。
3. **定根因**：区分是数据问题、代码问题、环境问题。
4. **做验证**：修复后补充失败用例并回归。

## Java 示例代码（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
class BizException extends RuntimeException {
    private final String code;

    BizException(String code, String message) {
        super(message);
        this.code = code;
    }

    String getCode() {
        return code;
    }
}

class UserRegisterService {
    int parseAge(String rawAge) {
        try {
            int age = Integer.parseInt(rawAge);
            if (age < 18 || age > 80) {
                // 业务边界：年龄不合规属于业务异常，不是系统异常
                throw new BizException("BIZ_AGE_RANGE", "age must be between 18 and 80");
            }
            return age;
        } catch (NumberFormatException ex) {
            // 底层异常转换成业务语义，避免上层暴露技术细节
            throw new BizException("BIZ_AGE_FORMAT", "age is not a valid number: " + rawAge);
        }
    }

    void register(String userName, String rawAge) {
        if (userName == null || userName.trim().isEmpty()) {
            throw new BizException("BIZ_NAME_EMPTY", "userName must not be empty");
        }
        int age = parseAge(rawAge);

        // 真实场景这里会写数据库，示例中只打印成功结果
        System.out.println("register success: userName=" + userName + ", age=" + age);
    }
}

public class Main {
    public static void main(String[] args) {
        UserRegisterService service = new UserRegisterService();

        // 三组输入：成功、业务范围错误、格式错误
        String[][] inputs = {
                {"alice", "20"},
                {"bob", "15"},
                {"carol", "abc"}
        };

        for (String[] input : inputs) {
            String userName = input[0];
            String age = input[1];
            try {
                service.register(userName, age);
            } catch (BizException ex) {
                // 统一输出错误码和语义，便于排查与监控聚合
                System.out.println("register failed: code=" + ex.getCode() + ", msg=" + ex.getMessage());
            }
        }
    }
}
```

**预期输出示例：**

```text
register success: userName=alice, age=20
register failed: code=BIZ_AGE_RANGE, msg=age must be between 18 and 80
register failed: code=BIZ_AGE_FORMAT, msg=age is not a valid number: abc
```

## 常见误区与修正

1. **误区：`catch (Exception)` 一把抓最省事**  
   修正：会掩盖错误语义，排障成本陡增。

2. **误区：吞异常不记录上下文**  
   修正：至少要记录错误码、关键参数、主键 ID。

3. **误区：不复现直接改代码**  
   修正：没有稳定复现，就无法验证修复是否有效。

## 面试高频问答

### Q1：什么时候该自定义业务异常？

- 一句话结论：当你需要对外提供稳定错误语义时，应使用业务异常。
- 关键点：
  1. 统一错误码便于前后端协作。
  2. 业务异常不应泄露底层实现细节。
  3. 可与监控告警系统直接打通。

### Q2：如何快速定位线上异常？

- 一句话结论：从“请求链路 + 输入参数 + 堆栈 + 最近变更”四个维度收敛问题。
- 关键点：
  1. 先定位第一抛出点，而不是最后 catch 点。
  2. 对比正常请求与异常请求输入差异。
  3. 修复后必须补失败用例回归。
