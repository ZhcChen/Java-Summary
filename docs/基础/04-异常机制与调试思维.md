# 04 异常机制与调试思维

## 本章目标

- 理解异常分类与处理边界。
- 写出可追踪、可定位、可恢复的异常逻辑。
- 形成“先定位根因，再修改代码”的调试方法。

## 前置知识

- 会使用 `try-catch`。
- 能阅读异常堆栈的基本结构。

## 核心概念

1. Checked Exception：编译期要求处理，常见于外部资源错误。
2. Runtime Exception：运行时错误，多与业务规则或代码缺陷相关。
3. 业务异常应表达业务语义（错误码 + 错误信息）。

## 原理展开

- 异常不是“报错即结束”，而是“边界内转换语义，边界外统一处理”。
- 盲目 `catch (Exception)` 并吞掉错误，会让排障成本指数上升。
- 调试要回到第一现场：输入、日志、堆栈、变更点。

## 典型场景

- 参数解析失败：转换为统一业务错误码。
- 下游超时：记录链路标识并做降级返回。
- 文件读取失败：明确资源错误来源。

## 常见误区

1. 吞异常不记录关键上下文。
2. 错误信息只有“失败了”，无业务标识。
3. 不复现直接改代码，导致问题反复。

## Java 示例代码（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
class BizException extends RuntimeException {
    private final String code;

    BizException(String code, String message) {
        super(message);
        this.code = code;
    }

    String getCode() {
        return code;
    }
}

public class Main {
    static int parsePositiveInt(String raw) {
        try {
            int value = Integer.parseInt(raw);
            if (value <= 0) {
                throw new BizException("BIZ_001", "value must be positive");
            }
            return value;
        } catch (NumberFormatException e) {
            // 转换为业务语义，避免上层直接暴露底层异常
            throw new BizException("BIZ_002", "invalid number: " + raw);
        }
    }

    public static void main(String[] args) {
        String[] inputs = {"12", "-2", "abc"};
        for (String input : inputs) {
            try {
                int result = parsePositiveInt(input);
                System.out.println("ok=" + result);
            } catch (BizException e) {
                System.out.println("error code=" + e.getCode() + ", msg=" + e.getMessage());
            }
        }
    }
}
```

**预期输出示例：**

```text
ok=12
error code=BIZ_001, msg=value must be positive
error code=BIZ_002, msg=invalid number: abc
```

## 练习与自测

1. 为“用户注册”设计 3 个业务异常码。
2. 写一份“异常定位四步法”并应用到一次真实错误。
3. 自测：能否说清“什么错误该捕获，什么错误应继续抛出”。

## 本章小结

- 异常处理是稳定性的基础，不是补丁式代码。
- 调试先复现与定位，再改动与验证。
