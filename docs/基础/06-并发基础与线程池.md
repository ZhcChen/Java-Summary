# 06 并发基础与线程池

## 本章目标

- 理解线程安全三要素：可见性、原子性、有序性。
- 区分 `volatile`、`synchronized`、原子类的适用边界。
- 掌握线程池参数与过载保护的基础思路。

## 前置知识

- 会写循环、方法、集合。
- 理解“多个任务同时执行”的基本概念。

## 一、并发问题本质

### 1) 三个高频关键词

- **可见性**：一个线程修改变量后，其他线程能否及时看到。
- **原子性**：复合操作是否不可分割（例如 `count++` 不是原子操作）。
- **有序性**：代码执行顺序是否符合预期（涉及重排序）。

### 2) 常见手段的边界

1. `volatile`：保证可见性与有序性，不保证复合操作原子性。
2. `synchronized`：提供互斥和可见性，适合保护临界区。
3. `AtomicInteger`：通过 CAS 保证单变量原子更新，性能通常优于粗粒度锁。

## 二、线程池不是“开线程”，是资源治理工具

### 1) 关键参数

- `corePoolSize`：核心线程数。
- `maximumPoolSize`：最大线程数。
- `workQueue`：任务队列（建议有界）。
- `RejectedExecutionHandler`：过载时拒绝策略。

### 2) 为什么需要有界队列

无界队列会把压力隐藏起来：短期看似平稳，长期会积压任务并放大延迟与内存风险。

## Java 示例代码（含注释，可直接运行）

**建议文件名：** `Main.java`  
**运行命令：** `javac Main.java && java Main`

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.RejectedExecutionException;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.AtomicInteger;

public class Main {
    private static int unsafeCounter = 0;
    private static int synchronizedCounter = 0;
    private static final AtomicInteger atomicCounter = new AtomicInteger(0);

    private static synchronized void increaseBySynchronized() {
        synchronizedCounter++;
    }

    public static void main(String[] args) throws InterruptedException {
        ThreadPoolExecutor pool = new ThreadPoolExecutor(
                4,
                8,
                30,
                TimeUnit.SECONDS,
                // 有界队列：避免任务无限堆积
                new ArrayBlockingQueue<>(100),
                // 明确拒绝策略：便于观测系统过载行为
                new ThreadPoolExecutor.AbortPolicy()
        );

        int submitCount = 5000;
        int rejected = 0;

        for (int i = 0; i < submitCount; i++) {
            try {
                pool.execute(() -> {
                    unsafeCounter++;                 // 非原子，可能丢更新
                    increaseBySynchronized();        // 加锁保证原子性
                    atomicCounter.incrementAndGet(); // CAS 保证原子更新
                });
            } catch (RejectedExecutionException ex) {
                rejected++;
            }
        }

        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);

        int accepted = submitCount - rejected;

        System.out.println("submitCount=" + submitCount);
        System.out.println("rejected=" + rejected);
        System.out.println("accepted=" + accepted);
        System.out.println("unsafeCounter=" + unsafeCounter);
        System.out.println("synchronizedCounter=" + synchronizedCounter);
        System.out.println("atomicCounter=" + atomicCounter.get());
    }
}
```

**预期输出示例：**

```text
submitCount=5000
rejected=...
accepted=...
unsafeCounter=...
synchronizedCounter=accepted
atomicCounter=accepted
```

> 说明：`unsafeCounter` 在并发下通常小于 `accepted`，具体数字每次可能不同。

## 常见误区与修正

1. **误区：`volatile` 能替代锁**  
   修正：`volatile` 不保证 `count++` 这类复合操作原子性。

2. **误区：线程池队列越大越安全**  
   修正：队列过大只是延迟问题暴露，不能解决根因。

3. **误区：只看吞吐，不看拒绝与延迟**  
   修正：并发治理必须同时关注成功率、延迟、资源占用。

## 面试高频问答

### Q1：`synchronized` 和 `AtomicInteger` 怎么选？

- 一句话结论：单变量计数优先原子类；复合临界区优先锁。
- 关键点：
  1. 原子类适合简单原子操作。
  2. 锁适合多变量一致性约束。
  3. 选择要看临界区复杂度。

### Q2：线程池为什么必须配置拒绝策略？

- 一句话结论：拒绝策略是过载保护开关，缺失会让系统行为不可控。
- 关键点：
  1. 高峰期任务增长必须可控。
  2. 拒绝行为可监控可预警。
  3. 便于上层做降级或限流。

## 练习与自测

1. 把示例中的 `AbortPolicy` 改为 `CallerRunsPolicy`，观察行为差异。
2. 在任务中加入 `Thread.sleep(5)`，感受队列积压与拒绝变化。
3. 解释：为什么线程池参数必须结合压测数据来定。

## 本章小结

- 并发编程先保证正确，再讨论性能。
- 线程池的核心价值是资源治理和过载控制。
