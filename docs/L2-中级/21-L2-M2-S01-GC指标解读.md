1. 先采集 GC 日志、线程栈、堆快照，再下结论。

## 术语解释（零基础友好）

- **GC**：垃圾回收机制。
- **OOM**：内存不足导致无法分配对象。

## 完整案例 Walkthrough（L2/L3 深挖）

### 场景输入

- 线上实例 Full GC 频率从每 30 分钟 1 次上升到每 2 分钟 1 次。

### 线上现象

- 接口抖动明显，CPU 占用高且 RT 阶段性飙升。

### 证据采集

- 收集 GC 日志、堆快照（heap dump）、线程栈（jstack）并关联业务高峰时段。

### 定位分析

- 定位为短命对象激增 + 部分对象晋升过快，老年代回收压力过大。

### 修复动作

- 优化对象创建路径，降低临时对象；微调堆分代参数并清理长生命周期缓存。

### 回归验证

- 对比优化前后 Full GC 次数、停顿时间、吞吐与错误率。

### 实战排障清单

- 证据优先，禁止“先拍参数后找原因”。
- 区分堆 OOM、元空间 OOM、直接内存问题。
- 调优后至少覆盖一次高峰流量窗口。


## 错答示例 -> 修正答法 -> 打分差异（章级题解）

### 练习题目（围绕本章：GC指标解读）

- 请用 90 秒说明“定义 -> 原理 -> 场景 -> 风险 -> 验证”完整答题链路。

### 常见错答示例（低分版）

- 只说概念，不说机制：例如只背定义，无法解释底层流程。
- 只说优点，不说边界：没有说明适用条件与失败场景。
- 没有指标验证：讲完方案后不给量化结果或回归口径。

### 修正答法（高分版）

1. 先给结论：一句话说清本章知识点解决什么问题。
2. 再讲原理：用 2~3 个关键机制串起完整流程。
3. 再落场景：给出一个可复现的业务场景和方案选择理由。
4. 再说风险：列出至少 2 个常见坑和对应防护动作。
5. 最后验证：给出可观测指标（如延迟、错误率、吞吐、资源占用）与目标阈值。

### 打分差异示例（同题对比）

| 评分维度 | 错答（低分） | 修正（高分） | 提升点 |
|---|---|---|---|
| 概念准确 | 只背术语 | 术语 + 边界条件 | 避免概念混淆 |
| 原理完整 | 断点式描述 | 链路化描述 | 解释能力更强 |
| 场景匹配 | 空泛举例 | 贴近业务约束 | 方案更可信 |
| 风险意识 | 不提失败 | 提供兜底与回滚 | 工程可落地 |
| 验证闭环 | 无量化指标 | 指标 + 阈值 + 回归 | 可复盘可验收 |

**运行命令：** `javac Main.java && java Main`

```java
public class Main {
    public static void main(String[] args) {
        int local = 1;               // 栈上局部变量
        byte[] bytes = new byte[256]; // 堆上对象
        // 类元数据位于方法区（JDK8+ 为元空间）
        System.out.println("local=" + local);
        System.out.println("heapSize=" + bytes.length);
    }
}
```
