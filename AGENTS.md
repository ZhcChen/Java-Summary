# AGENTS

## 项目目标
- 面向 Java 学习与工程实践，构建「初级 -> 中级 -> 高级」可递进的知识汇总。
- 同时适配 0 基础学习者与有经验开发者，不局限于求职场景。
- 内容强调：清晰、易懂、可复习、可面试表达。
- 每个主题都应兼顾：概念、原理、场景、面试问答、示例代码。

## 语言与风格
- 默认使用简体中文；技术术语保留英文（如 JVM、GC、AOP、Redis）。
- 先结论后展开，段落短、标题明确，避免大段堆砌。
- 面试题回答优先使用「一句话结论 + 关键点 3~5 条 + 场景补充」结构。
- 以“写一本可系统学习的技术书”为标准组织内容，优先保证结构清晰、叙述完整，不追求花哨排版。
- 避免堆砌过多图表与装饰性结构；图表只在确实能降低理解成本时使用。

## 文档组织约定
- 总览路线：`/README.md`、`/docs/00-总览-学习路线.md`
- 总索引入口：`/docs/README.md`
- 学习索引：`/docs/01-按学习顺序索引.md`
- 面试索引：`/docs/02-按面试频率索引.md`
- 专题索引：`/docs/03-按专题索引.md`
- 初级：`/docs/L1-初级/`
- 中级：`/docs/L2-中级/`
- 高级：`/docs/L3-高级/`
- 示例代码：`/examples/`
- 新增主题时，优先补齐对应层级目录中的 README 索引。

## 索引文档规范（重点）
- 索引遵循「总入口 -> 维度索引 -> 分层索引 -> 主题文档」四级结构。
- 维度索引至少包含：按学习顺序、按面试频率、按专题。
- 文件名使用编号前缀（如 `01-`、`02-`）保证 GitHub 展示顺序稳定。
- 索引表格建议统一字段：主题 / 必会度 / 面试频率 / 状态。
- 状态统一使用：`TODO`、`DOING`、`DONE`，便于维护进度。
- 每条索引必须能直达本仓库内文档，不依赖外部项目链接。
- 索引条目至少包含：先修知识、学习目标、对应正文、对应练习或代码入口。
- 索引优先“纯导航”原则：少解释、少修饰、少压力信息，只保留必要跳转路径。

## 单篇文档模板（建议）
每篇文档建议包含以下小节：
1. 这是什么（定义）
2. 为什么重要（价值）
3. 怎么工作（原理）
4. 典型场景（何时用）
5. 常见误区（边界）
6. 面试高频问答（含标准答法）
7. 示例代码（最小可运行）
8. 延伸阅读（官方文档或权威资料）

## 0 基础详细度标准（强制）
- 每篇正文必须明确“前置知识”（不会什么先补什么）。
- 每篇正文必须包含至少 1 个可运行 Java 示例，并给出预期输出。
- 每篇正文必须包含“常见错误”与“复习检查/练习任务”，形成闭环。
- 术语首次出现时尽量给一句解释，避免仅用黑话堆叠。

## 图表规范（Markdown + Mermaid）
- 优先使用 Mermaid：`flowchart`、`sequenceDiagram`、`classDiagram`、`mindmap`。
- 图只表达一个核心逻辑，节点命名短且语义明确。
- 图前写 1~2 行背景说明，图后补充关键结论。

## 示例代码规范
- 示例代码优先小而完整，可直接复制运行。
- 单个示例只演示一个关键点；复杂场景拆分多个文件。
- 代码中加入必要注释，说明关键机制与易错点。
- 示例代码注释要求“解释原因”，而不是仅解释语法。
- 代码块需同时给出：建议文件名、运行命令、预期输出。

## 引用与链接规范（强制）
- 学习主链路文档默认不引用外部项目仓库链接。
- 延伸阅读优先引用本仓库内已整理文档；确需外部资料时，单独说明原因并经用户确认。

## 正确性与质量门槛
- 涉及版本差异（如 JDK8/11/17、Spring Boot 2/3）必须显式标注。
- 涉及性能结论时，避免绝对化描述，给出适用前提。
- 涉及面试题答案时，尽量给出「为什么」，不仅是「是什么」。
- 引用外部资料时，优先官方文档与权威来源。

## 贡献与维护
- 任何新增内容，至少包含：概念说明 + 1 个面试问答 + 1 段示例代码或图。
- 大章节优先先补索引，再逐步扩展，避免“有目录没内容”。

## 提交与同步规则（强制）
- 只要文档或代码有变动，必须在本地完成 `git add` + `git commit`。
- 每次提交后必须立即 `git push` 到远程 `main` 分支，确保 GitHub 可见最新内容。
- 禁止累计大量未提交改动；建议按“一个主题/一次调整”形成小步提交。
